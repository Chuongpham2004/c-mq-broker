1. Introduction

1.1 Problem Statement

Modern distributed systems rely heavily on asynchronous message passing to decouple services, buffer workloads, and enable event-driven architectures. Production message brokers such as Apache Kafka, RabbitMQ, and MQTT-based systems manage millions of messages per second, yet their internal mechanics—memory-mapped persistence, acknowledge-based delivery guarantees, and topic hierarchy routing—are rarely explored at the systems-programming level.

This project addresses the challenge of building a fully functional, topic-based Publish-Subscribe message broker from scratch in C, relying exclusively on the C11 standard library and POSIX system calls. The goal is to demonstrate mastery of core systems programming concepts: TCP socket programming, multi-threaded concurrency with mutexes, memory-mapped file I/O, signal handling, and durable persistence under crash scenarios.

1.2 Objectives

The project implements the following eight features:

  1. Topic-based pub-sub with PUBLISH, SUBSCRIBE, UNSUBSCRIBE, and ACK commands.
  2. Persistent append-only log per topic that survives broker restarts.
  3. Configurable fsync policy: per-write, group-commit, or time-based.
  4. Per-subscriber cursors (persisted to disk) for resume-on-reconnect.
  5. Pending-ACK list with automatic redelivery on disconnect or timeout.
  6. Topic hierarchy matching: + (single level) and # (multi-level) wildcards.
  7. Retention policies: time-based, size-based, and all-acknowledged.
  8. Durability test: kill -9 the broker mid-publish, restart, and verify zero data loss.

1.3 Approach Overview

We designed a layered architecture separating concerns into five modules: networking (TCP I/O and protocol parsing), thread pool (Boss-Worker concurrency), broker core (pub-sub logic, wildcards, sessions, ACK management), storage engine (mmap-based append-only writes with configurable fsync), and background workers (redelivery and retention). All inter-thread communication is synchronized using POSIX mutexes with a strict lock-ordering discipline to prevent deadlocks. The system compiles cleanly with gcc -Wall -Wextra -std=c11 producing zero warnings, and all five automated test cases pass consistently.

2. Background & Theory

2.1 Publish-Subscribe Messaging Model

The Publish-Subscribe (Pub-Sub) pattern decouples message producers from consumers through an intermediary called a broker. Producers publish messages to named channels called topics, and consumers subscribe to topics of interest. The broker is responsible for fan-out: routing each published message to all subscribers whose subscription pattern matches the message's topic. This model enables one-to-many communication without producers needing awareness of consumer identities or counts [1].

Topic-based systems often support hierarchical topic names using a delimiter (typically /), allowing wildcard subscriptions. The MQTT specification [2] defines two wildcard characters: + matches exactly one level (e.g., sensor/+/temp matches sensor/room1/temp but not sensor/room1/room2/temp), and # matches zero or more levels (e.g., sensor/# matches sensor, sensor/temp, and sensor/a/b/c).

2.2 Memory-Mapped I/O (mmap)

The mmap() system call maps a file's contents directly into the process's virtual address space, enabling file I/O through simple pointer arithmetic and memcpy() instead of explicit read()/write() system calls [3]. With MAP_SHARED, modifications to the mapped region are automatically reflected in the underlying file. The msync() call can be used to explicitly flush dirty pages to disk: MS_SYNC blocks until the write is confirmed on the storage device, while MS_ASYNC initiates a non-blocking writeback. This mechanism gives developers fine-grained control over the durability vs. performance trade-off.

2.3 POSIX Threads and Synchronization

POSIX Threads (pthreads) provide the standard API for multi-threaded programming on Linux [4]. Our system uses multiple mutex types to protect shared state: a global lock for the topic registry, per-topic locks for subscriber lists, and dedicated locks for the pending-ACK queue, cursor records, and message ID counter. A consistent lock-ordering policy prevents deadlock: the global lock is always acquired before topic locks, which are acquired before the pending-ACK lock and storage mutex.

2.4 Signal Handling in Multi-Threaded Programs

In multi-threaded programs, signals are delivered to an arbitrary thread unless explicitly directed. We use sigaction() with SA_RESTART intentionally disabled so that blocking calls such as accept() return immediately with errno = EINTR when SIGINT or SIGTERM is received. The signal handler sets a volatile sig_atomic_t flag that the main accept loop checks on each iteration, enabling a clean and deterministic shutdown sequence [5].

3.3 Binary Protocol Design

All client-broker communication uses a compact binary protocol. Every message consists of a fixed 14-byte header followed by variable-length topic and payload fields:

Table 2: MessageHeader layout (14 bytes, packed, no padding)
FieldTypeSizeDescription
typeuint8_t1 BMessage type: CONNECT(1), PUBLISH(2), SUBSCRIBE(3), ACK(4), UNSUBSCRIBE(5)
topic_lenuint8_t1 BLength of topic string (max 255 bytes)
msg_iduint32_t4 BUnique message identifier (network byte order)
timestampuint32_t4 BUnix epoch timestamp (network byte order)
payload_lenuint32_t4 BPayload length in bytes (network byte order)

The __attribute__((packed)) directive ensures the compiler inserts no padding, guaranteeing exactly 14 bytes on all platforms. Multi-byte fields are transmitted in network byte order (big-endian) using htonl()/ntohl().

typedef struct __attribute__((packed)) {
    uint8_t  type;
    uint8_t  topic_len;
    uint32_t msg_id;
    uint32_t timestamp;
    uint32_t payload_len;
} MessageHeader;  // Total: 14 bytes

4. Implementation

4.1 Persistent Append-Only Storage

Each topic maps to a dedicated log file on disk under the ./data/ directory. The topic name's / delimiters are replaced with _ to form a valid filename (e.g., sensor/nhietdo → data/sensor_nhietdo.log). Messages are appended sequentially using a five-step mmap-based process: 

size_t storage_save_message(const char *topic, Message *msg) {
    // 1. Open file (O_RDWR | O_CREAT)
    int fd = open(filepath, O_RDWR | O_CREAT, 0666);

    // 2. Get current file size
    struct stat st;
    fstat(fd, &st);
    size_t old_size = st.st_size;
    size_t new_size = old_size + record_size;

    // 3. Extend file to accommodate new record
    ftruncate(fd, new_size);

    // 4. Memory-map the entire file
    void *map = mmap(NULL, new_size, PROT_READ | PROT_WRITE,
                     MAP_SHARED, fd, 0);

    // 5. Write header + topic + payload at end
    memcpy(map + old_size, &header, sizeof(MessageHeader));
    memcpy(map + old_size + sizeof(header), topic, topic_len);
    memcpy(map + old_size + sizeof(header) + topic_len,
           payload, payload_len);

    // 6. Fsync according to configured policy
    msync(map, new_size, MS_SYNC);  // or MS_ASYNC

    munmap(map, new_size);
    close(fd);
    return new_size;  // end offset = next cursor position
}

This design achieves zero-copy persistence: the kernel's page cache handles the actual disk write, eliminating an extra copy into a userspace buffer. The on-disk format mirrors the wire format identically (14-byte header + topic + payload), so replay reads can send data directly to clients without re-serialization.

4.3 Topic Wildcard Matching

The match_topic(sub, pub) function performs character-by-character comparison between a subscription pattern and a published topic name, handling two wildcard characters: 

int match_topic(const char *sub, const char *pub) {
    const char *s = sub, *p = pub;
    while (1) {
        if (*s == '#') return 1;  // # matches everything remaining
        if (*s == '+') {
            // Skip exactly one level in the published topic
            while (*p != '/' && *p != '\0') p++;
            s++;
            continue;
        }
        if (*s == *p) {
            if (*s == '\0') return 1;  // exact match
            s++; p++;
        } else {
            // Edge case: "sport/#" matches "sport"
            if (*s == '/' && *(s+1) == '#' && *(s+2) == '\0'
                && *p == '\0')
                return 1;
            return 0;
        }
    }
}

4.4 Fan-Out with Deduplication

During publish, the broker iterates over all registered topics, checking each subscription pattern against the published topic using match_topic(). A deduplication array (sent_sockets[]) prevents a consumer who subscribes to multiple matching patterns from receiving duplicate copies of the same message: 

int sent_sockets[MAX_SUBSCRIBERS * MAX_TOPICS];
int sent_count = 0;

for (int idx = 0; idx < broker.topic_count; idx++) {
    if (match_topic(broker.topics[idx].name, msg->topic)) {
        for (int i = 0; i < topic->sub_count; i++) {
            int sub_sock = topic->subscriber_sockets[i];
            // Skip if already sent to this socket
            int already_sent = 0;
            for (int s = 0; s < sent_count; s++)
                if (sent_sockets[s] == sub_sock) { already_sent = 1; break; }
            if (already_sent) continue;
            sent_sockets[sent_count++] = sub_sock;
            // ... send message + create PendingAck
        }
    }
}

4.5 Per-Subscriber Cursors & Replay

Each subscriber's read position is tracked as a byte offset into the corresponding topic's log file. The cursor is updated only on successful ACK, and it advances forward-only (a new offset must exceed the current one). Cursors are persisted in data/cursors.txt with format client_id:topic:offset per line, reloaded on broker startup via load_cursors().

When a client subscribes, the function replay_missed_messages() scans all .log files in data/, matches each file's topic against the subscription pattern, and replays all messages from the cursor offset to the end of file: 

// Simplified replay logic:
size_t cursor = get_cursor(client_id, orig_topic);
void *map = mmap(NULL, file_size, PROT_READ, MAP_SHARED, fd, 0);
size_t offset = cursor;
while (offset + sizeof(MessageHeader) <= file_size) {
    MessageHeader *h = (MessageHeader *)((char *)map + offset);
    size_t record_size = sizeof(MessageHeader) + h->topic_len
                       + ntohl(h->payload_len);
    // Send record to client, create PendingAck
    send(client_socket, &header, sizeof(MessageHeader), 0);
    send(client_socket, orig_topic, topic_len, 0);
    send(client_socket, payload, payload_len, 0);
    offset += record_size;
}

4.6 Pending-ACK & Redelivery

Every message sent to a consumer creates a PendingAck node in a singly-linked list protected by pending_ack_lock. The background redelivery_worker thread runs every 1 second and scans this list. If any entry has been unacknowledged for ≥5 seconds, it is retransmitted. After 3 failed retries, the broker forcibly disconnects the client using shutdown(fd, SHUT_RDWR) followed by close(fd). The shutdown() call is critical: it unblocks any thread currently in read_exact() on that socket, preventing thread hangs.

4.7 Retention Policies

The retention_worker thread runs every 5 seconds and processes each .log file against three configurable policies:

  1. Time-based (--retention-time N): Removes messages with timestamp < now - N.
  2. Size-based (--retention-size N): Trims from the front when file size exceeds N bytes.
  3. All-ACKed (--retention-acked): Removes messages acknowledged by all active subscribers (offset ≤ minimum cursor).

When multiple policies are active, the maximum trim offset among all three is used, ensuring the most aggressive cleanup. After trimming, cursor offsets and pending-ACK file offsets are adjusted downward by the number of removed bytes to maintain correctness.

The physical trimming is performed by trim_log_file(), which uses mmap + memmove + ftruncate to shift remaining data to the front of the file.

4.8 Thread Pool (Boss-Worker)

The thread pool implements the Boss-Worker pattern: the main thread (boss) accepts TCP connections and enqueues them into a circular buffer of size 100. Four pre-spawned worker threads block on a condition variable, waking up to dequeue and process client sessions. Each worker handles one client session for its entire lifetime (connect → publish/subscribe loop → disconnect). The pool uses a single mutex and condition variable, avoiding lock contention for the critical accept-dispatch path.

5. Testing & Validation

5.1 Test Methodology

We built an automated test suite (test_suite.sh, 176 lines) that runs via make test. The suite starts a fresh broker instance for each test case, exercises specific functionality using the client programs, captures output in log files, and checks for expected patterns using grep. Each test case reports PASS or FAIL with clear formatting. The suite covers 5 distinct test cases spanning normal operations, edge cases, and error conditions.

Uses noack_client which deliberately never sends ACK. The test waits 22 seconds (enough for 3 retries at 5-second intervals plus margin) and verifies: (a) the client received ≥3 copies of the message (original + retries), and (b) the broker disconnected the client after exceeding the retry limit.

5.3 Durability Test (kill -9)

A separate durability test demonstrates crash resilience. The broker is started with --fsync per-write, a producer sends messages, and the broker is killed with kill -9 (SIGKILL, which cannot be caught or handled). After restart, a consumer reconnects and verifies that all previously published messages are replayed from the intact log files on disk. This works because:

  1. ftruncate() extends the file before data is written.
  2. memcpy() into a MAP_SHARED region writes directly to the kernel page cache.
  3. msync(MS_SYNC) flushes the page cache to the physical storage device before returning.

6. Performance Analysis

6.1 Fsync Policy Benchmarks

We measured message throughput by publishing 1,000 messages of 100 bytes each to a single topic under each fsync policy. The broker was run on a standard Linux x86-64 system with an SSD.

Table 7: Throughput comparison across fsync policies (1,000 messages × 100 bytes)
PolicyMessages/secRelative SpeedData at Risk on Crash
per-write~8001.0×0 messages
group-commit~3,2004.0×Up to 4 messages
time-based~5,5006.9×Up to ~1s of messages

The per-write policy is I/O-bound: each msync(MS_SYNC) call blocks until the storage controller confirms the write. The group-commit policy amortizes this cost over 5 messages, achieving a 4× speedup. The time-based policy performs best under burst workloads, as multiple messages within a 1-second window share a single sync.

6.2 Bottleneck Analysis

6.2.1 I/O Bottleneck (Storage)

The primary bottleneck is msync(MS_SYNC) in per-write mode. Each call forces a full flush to disk, taking 0.5–2ms on SSDs and 5–15ms on HDDs. This dominates the per-message processing time. The storage_mutex further serializes all writes, meaning only one message can be persisted at a time regardless of thread pool size.

6.2.2 Fan-Out Complexity

The wildcard matching in broker_publish() has O(T × S) complexity, where T is the number of registered topics and S is the maximum subscriber count per topic. With the current limits of T=64 and S=32, the maximum fan-out scan is 2,048 comparisons, which completes in microseconds and is not a practical bottleneck.

6.2.3 Cursor Persistence

Cursors are saved to cursors.txt on every ACK by rewriting the entire file. For high-throughput scenarios with many subscribers, this could become a bottleneck. A future optimization would use an append-only cursor log or a binary format with in-place updates.

7. Conclusion & Future Work

7.1 Summary

We successfully designed and implemented C-MQ Broker, a complete topic-based Publish-Subscribe message broker in 2,109 lines of C11 code using only the POSIX standard library. The system delivers all eight required features: five pub-sub commands, persistent append-only storage, three configurable fsync policies, per-subscriber cursor tracking with resume-on-reconnect, pending-ACK management with automatic redelivery, MQTT-style wildcard topic matching, three retention policies, and verified crash durability under kill -9. All five automated test cases pass consistently.

7.2 Lessons Learned

  • mmap complexity: While mmap provides excellent performance, managing file extensions with ftruncate, proper munmap, and concurrent access requires careful coding. Forgetting to munmap before close can lead to data corruption.
  • Signal safety: In multi-threaded programs, signal handlers must be extremely minimal (only set a flag). Any complex work in a signal handler risks deadlock if the interrupted thread held a mutex.
  • Lock ordering: Without a documented lock ordering policy, we encountered deadlocks during development when the redelivery worker acquired pending_ack_lock while a publish thread held global_lock and tried to acquire pending_ack_lock in reverse order.
  • Socket shutdown semantics: Using close() alone on a socket that another thread is reading from does not reliably unblock that thread on all platforms. The shutdown(SHUT_RDWR) call is necessary to force an immediate error return from blocking recv()/read() calls.

7.3 What We Would Do Differently

  • Replace fixed-size arrays (MAX_TOPICS=64, MAX_SUBSCRIBERS=32) with dynamically-sized hash maps for better scalability.
  • Use a binary format for cursor persistence instead of text-based cursors.txt to reduce I/O overhead.
  • Implement per-topic storage file descriptors cached in memory, avoiding repeated open()/close() on each publish.
  • Add proper logging with severity levels instead of printf statements.

7.4 Future Work

  • TLS/SSL encryption: Integrate OpenSSL for encrypted client-broker communication.
  • Authentication: Add username/password or token-based authentication on CONNECT.
  • Clustering: Implement broker federation or replication for horizontal scaling.
  • QoS levels: Support MQTT-like QoS 0 (at-most-once) and QoS 2 (exactly-once) in addition to current QoS 1 (at-least-once).
  • WebSocket support: Enable browser-based clients to connect directly to the broker.
